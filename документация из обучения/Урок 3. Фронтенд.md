**Страница 2**

Жизненный цикл загрузки веб-страницы

Понимание того, как браузер превращает код в живую страницу, — ключ к созданию быстрых и эффективных приложений. Давайте разберем этот процесс пошагово.

01 Запрос к серверу
Пользователь вводит URL или кликает ссылку, браузер отправляет HTTP-запрос на сервер, который начинает обработку запроса и подготовку необходимых файлов.

02 Получение файлов
Сервер возвращает HTML-файл (первые 14КВ в первом пакете), который содержит ссылки на CSS и JavaScript. Браузер начинает параллельную загрузку этих ресурсов.

03 Построение DOM
Браузер построчно читает HTML (токенизация) и создает DOM — древовидную структуру документа, представляющую иерархию всех элементов страницы.

04 Построение CSSOM
Параллельно парсятся CSS-файлы и создается CSSOM — дерево стилей. Важно: CSS блокирует рендеринг! Браузер не покажет страницу до загрузки стилей.

05 Создание Render Tree
Браузер объединяет DOM и CSSOM, создавая Render Tree — структуру только видимых элементов (элементы с display: none исключаются).

06 Layout и Painting
Рассчитываются точные размеры и позиции (Layout/Reflow), затем элементы преобразуются в реальные пиксели с применением цветов, изображений и эффектов (Painting).

07 Compositing
Все отрисованные слои собираются вместе и отображается финальная страница. При изменениях через JavaScript процесс Layout → Paint → Composite повторяется.

**Страница 3**

Critical Rendering Path: оптимизация загрузки

Critical Rendering Path — это последовательность шагов, которые браузер должен выполнить для отображения страницы. Понимание этого процесса критически важно для оптимизации производительности.

Путь критического рендеринга

HTML → DOM
Парсинг HTML и построение объектной модели документа

CSS → CSSOM
Обработка стилей и создание дерева стилей

DOM + CSSOM → Render Tree
Объединение структуры и стилей

Layout → Paint → Composite
Расчет позиций, отрисовка и сборка финальной страницы

Ключевые метрики производительности

LCP (Largest Contentful Paint): < 2.5s — время загрузки основного контента
FID (First Input Delay): < 100ms — время до первой возможности взаимодействия
CLS (Cumulative Layout Shift): < 0.1 — стабильность визуальной компоновки
TTFB (Time to First Byte): < 600ms — скорость ответа сервера

Секреты оптимизации

Минимизация файлов: сжимайте HTML, CSS и JavaScript для уменьшения размера передаваемых данных.

Критические стили inline: встраивайте основные стили прямо в HTML для мгновенного отображения.

async и defer: используйте эти атрибуты для скриптов, чтобы не блокировать парсинг HTML.

Оптимизация изображений: используйте WebP/AVIF форматы и ленивую загрузку.

CDN: распределенная доставка контента ускоряет загрузку по всему миру.

**Страница 4**

Типы рендеринга: выбираем стратегию

Современная веб-разработка предлагает несколько подходов к рендерингу страниц. Каждый метод имеет свои преимущества и оптимален для определенных сценариев использования.

CSR — Client-Side Rendering
Рендеринг на стороне клиента
Сервер отправляет почти пустой HTML + большой JS-файл. JavaScript в браузере создает весь интерфейс.
Плюсы: быстрая навигация после загрузки, высокая интерактивность, идеально для SPA
Минусы: медленная первая загрузка, слабое SEO, требует JavaScript
Используйте для: дашбордов, админ-панелей, веб-приложений без требований к SEO

SSR — Server-Side Rendering
Рендеринг на стороне сервера
Сервер генерирует полный HTML для каждого запроса. Браузер получает готовую страницу с контентом.
Плюсы: быстрая первая отрисовка, отличное SEO, работает без JavaScript
Минусы: нагрузка на сервер, медленнее переходы между страницами
Используйте для: новостных сайтов, e-commerce, блогов с частыми обновлениями

SSG — Static Site Generation
Статическая генерация
Страницы генерируются при сборке проекта. Сервер отдает готовые HTML-файлы через CDN.
Плюсы: максимально быстрая загрузка, лучшее SEO, низкая стоимость хостинга
Минусы: нужна пересборка для обновлений, не подходит для динамических данных
Используйте для: блогов, портфолио, документации, лендингов

ISR — Incremental Static Regeneration
Инкрементальная регенерация
Комбинация SSG и SSR — статические страницы обновляются по расписанию в фоновом режиме.
Плюсы: скорость SSG + актуальность SSR, масштабируемость
Минусы: сложность настройки, зависимость от фреймворка
Используйте для: e-commerce с редкими обновлениями товаров, новостных сайтов

Современные фреймворки как Next.js и Nuxt.js позволяют комбинировать разные стратегии рендеринга в рамках одного проекта — это гибридный подход, где разные страницы используют оптимальную для них технологию.

Ключевые концепции профессионального фронтенда

SEO — поисковая оптимизация

Ваш сайт должен быть виден поисковикам. Ключевые практики:

Семантический HTML: используйте header, nav, main, article, section
Meta-теги: title (до 60 символов), description (до 160 символов)
Structured Data: Schema.org разметка для богатых сниппетов
Core Web Vitals: LCP < 2.5s, FID < 100ms, CLS < 0.1
URL структура: /blog/frontend-guide ✅ vs /article?id=123 ❌
Alt-атрибуты: описывайте изображения для доступности и SEO

SSR/SSG лучше для SEO, чем чистый CSR, так как контент сразу в HTML.

UI — пользовательский интерфейс

Визуальный дизайн вашего приложения. Принципы:

Визуальная иерархия: размер и контраст показывают важность
Консистентность: единый стиль во всем приложении
Читаемость: контраст 4.5:1 минимум, шрифт от 16px
Responsive: mobile-first дизайн обязателен

Популярные решения 2025: Tailwind CSS (utility-first), CSS Modules, Styled-components.

UX — пользовательский опыт

Как пользователь ощущает взаимодействие с продуктом:

Jakob's Law: пользователи ожидают привычных паттернов
Fitts's Law: важные кнопки крупнее и ближе
Doherty Threshold: интерфейс реагирует < 400ms
Не заставляйте думать: интуитивная навигация
Feedback: визуальный отклик на действия
AI-персонализация: адаптация под пользователя

С попощью овощей персонализации — используйте данные о поведении для улучшения UX.

Accessibility (A11y)

15% населения имеют ограниченные возможности:

Семантический HTML вместо div-супа
Keyboard navigation: Tab, Enter, Space
ARIA-атрибуты: aria-label, role, aria-live
Цветовой контраст: минимум 4.5:1
Screen readers: тестируйте с NVDA, VoiceOver

Инструменты: axe DevTools, Lighthouse, Wave.

Безопасность фронтенда

Защита пользовательских данных критически важна:

XSS (Cross-Site Scripting): экранируйте пользовательский ввод. React/Vue делают это автоматически. Используйте DOMPurify для HTML.

CSRF (Cross-Site Request Forgery): CSRF-токены, SameSite cookies, custom headers.

Аутентификация: JWT в httpOnly cookies (НЕ в localStorage!), OAuth 2.0, 2FA.

HTTPS обязательно для всех сайтов — это SEO-фактор.

Регулярно проверяйте зависимости: npm audit

Производительность

Lazy Loading: изображения и компоненты

Минификация: HTML, CSS, JS

**Страница 8**

Что такое "состояние" и почему оно важно для твоего приложения?

Представь, что у любого объекта вокруг нас есть "состояние": выключатель может быть "включен" или "выключен", светофор может быть "красным" или "зеленым". Это то, как объект выглядит или ведет себя в определенный момент.

В веб-приложении "состояние" – это все данные, которые оно помнит и использует прямо сейчас. Например:

Ты вошел на сайт? (состояние: "авторизован")
Сколько товаров в твоей корзине? (состояние: "3 товара")
Включена ли темная тема? (состояние: "темная тема активна")
Открыто ли выпадающее меню? (состояние: "меню открыто")

Управление этим "состоянием" – это умение правильно хранить, изменять и показывать эти данные пользователю. Без этого приложение быстро станет запутанным, медленным и сложным для понимания.

Помощники для работы с состоянием (State Management)

Когда приложение становится большим, управлять всеми данными вручную очень сложно. Для этого существуют специальные библиотеки:

Zustand
Простой и быстрый. Отличный выбор для большинства проектов, когда нужно быстро и эффективно управлять состоянием.

Redux Toolkit
Мощный и структурированный. Подходит для очень больших приложений со сложной логикой, где нужна строгая организация данных.

Jotai
"Атомарный" подход. Идеален, когда нужно очень тонко управлять небольшими частями состояния, чтобы приложение работало максимально быстро.

Context API
Встроен прямо в React. Удобно для простых вещей, как, например, переключение темы или статус авторизации. Но для большого и часто меняющегося состояния может работать не очень быстро.

Что такое React Query / TanStack Query? Это не замена вышеперечисленным. Это инструмент для работы с данными, которые приходят с сервера (например, список товаров, информация о пользователе). Он умеет их кэшировать, автоматически обновлять и показывать пользователю самые актуальные данные.

Разделение логики: где что должно быть?

Важно понимать, какие задачи должен выполнять твой веб-сайт (фронтенд), а какие — сервер (бэкенд).

Задачи фронтенда (на твоем компьютере) ✅

Отображение интерфейса: Открыть меню, показать всплывающее окно, сделать анимацию.
Простые проверки: Проверить, что поле формы не пустое или что email введен правильно (до отправки на сервер).
Локальные настройки: Твоя выбранная тема (темная/светлая), язык.
Навигация: Переключение между страницами сайта.
Легкие расчеты: Посчитать, сколько стоит товар, если ты добавил 2 штуки.

Золотое правило: на фронтенде оставляй то, что улучшает удобство, не требует доступа к базам данных и не связано с критической безопасностью.

Задачи бэкенда (на сервере) ⚠️

Самые важные проверки: Настоящая проверка всех данных, которые ты ввел (например, есть ли такой пользователь, корректен ли номер карты).
Бизнес-логика: Расчет скидок, генерация отчетов, управление заказами.
Работа с данными: Сохранение, изменение или удаление информации в базе данных.
Безопасность: Подтверждение, что ты – это ты (авторизация), и у тебя есть права на те или иные действия.
Деньги: Обработка платежей.

⚠️ Золотое правило безопасности
НИКОГДА НЕ ДОВЕРЯЙ ФРОНТЕНДУ! Пользователь может обойти любые проверки на своем компьютере. Поэтому всегда перепроверяй все важные данные на сервере, особенно те, что связаны с деньгами, безопасностью или данными других пользователей.

**Страница 11**

Мастерство промптинга для фронтенда

Качество результата AI напрямую зависит от качества вашего промпта. Научитесь правильно формулировать запросы — и AI станет вашим супер-инструментом.

1. Четкая цель и контекст

❌ Плохо: "Создай форму"
✅ Хорошо: "Создай React-форму регистрации с полями email, пароль, подтверждение пароля. Валидация на стороне клиента. Используй React Hook Form + Zod. Стили — Tailwind CSS."

Всегда указывайте конкретные требования, технологии и желаемое поведение компонента.

2. Укажите формат возврата

"Создай компонент Button со следующими требованиями:
— Варианты: primary, secondary, outline
— Размеры: sm, md, lg
— Disabled состояние
— Loading состояние со спиннером
— TypeScript типы
Верни готовый код с примерами использования."

Структурированный запрос = структурированный результат.

3. Технический стек явно

"Стек проекта:
— React 18 + TypeScript
— Vite
— Tailwind CSS + shadcn/ui
— React Hook Form
— Zustand для state
— React Query для API
Создай компонент UserProfile..."

AI адаптирует код под ваш конкретный стек.

4. Визуальные референсы

Прикрепляйте скриншоты желаемого дизайна. GPT-5 и Claude отлично понимают UI по изображениям. Можно загружать Figma mockups или фото интерфейсов.

"Вот скриншот дизайна [прикрепить]. Создай похожий компонент с такой же структурой и стилями."

5. Примеры существующего кода

"Вот пример нашего компонента:
[код существующего компонента]
Создай аналогичный компонент для [новая функция], следуя той же структуре, стилю кодирования и паттернам."

Это обеспечивает консистентность кодовой базы.

Последовательные промпты (Sequential Prompting)

1. "Создай структуру React-компонента для дашборда аналитики"
2. [AI создает структуру] → "Добавь графики с использованием Recharts"
3. [AI добавляет графики] → "Добавь фильтры по дате и экспорт в CSV"
4. [AI добавляет функции] → "Оптимизируй производительность"

Итеративное улучшение дает лучшие результаты, чем один гигантский промпт.

Промпт для рефакторинга

"Вот мой компонент [вставить код]. Проблемы: слишком большой (200+ строк), много useState, нет TypeScript типов, неоптимальный рендеринг.

Задача:
1. Разбей на подкомпоненты
2. Вынеси логику в custom hooks
3. Добавь TypeScript типы
4. Оптимизируй (memo, useMemo)
5. Объясни каждое изменение"
