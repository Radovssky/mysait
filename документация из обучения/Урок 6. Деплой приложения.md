# Урок 6: Деплой приложения — от Docker до продакшна

## 1. Введение в Docker для вайбкодинга

### Что такое Docker простыми словами

Docker — это контейнер для вашего приложения, где упаковано всё необходимое: код, зависимости, настройки операционной системы. Это как коробка, которую можно открыть на любом компьютере или сервере, и всё будет работать одинаково.

Представьте: вы разрабатывали приложение на Mac с Node.js 18, PostgreSQL 14 и Redis. На вашем компьютере всё идеально работает. Но когда вы копируете код на сервер Ubuntu — вдруг там Node.js 16, PostgreSQL 12, и ничего не работает. Docker решает эту проблему: вы упаковываете Node.js 18, PostgreSQL 14 и Redis в контейнер, и этот контейнер работает везде одинаково.

### Почему Docker критически важен для новичка в вайбкодинге

**Проблема без Docker:**
- Разрабатываете локально → "У меня работает!"
- Копируете на сервер → Crash, ошибки, часы отладки
- Обновления система → Приложение ломается

**Решение с Docker:**
- Всё точно так же локально и на сервере
- Нет разницы между компьютерами и окружениями
- Масштабирование: нужно больше мощности? Просто запустите ещё контейнер
- Вайбкодинг работает лучше: AI может генерировать Dockerfile, и вы не погружаетесь в детали

### Архитектура Docker простыми словами

- **Dockerfile** — рецепт для сборки контейнера. Описывает, что нужно установить и как запустить приложение.
- **Image (образ)** — готовый "снимок" приложения со всеми зависимостями. Размер: 200MB-2GB.
- **Container (контейнер)** — запущенный образ. Это как экземпляр приложения, которое работает на сервере. Из одного образа можно запустить 100 контейнеров.
- **Docker Compose** — инструмент для запуска нескольких контейнеров вместе (например, приложение + база данных + Redis).
- **Registry (хранилище)** — место, где хранятся образы. Docker Hub — публичное хранилище, как GitHub для Docker.

## 2. Глубокое погружение в Docker и Docker-низацию

### Зачем Dockerизировать приложение?

**Основные причины:**

1. **Одинаковое окружение везде** — Локально: Node.js 18, PostgreSQL 14, Redis 7. На сервере: Node.js 18, PostgreSQL 14, Redis 7. Без Docker: мучитесь с версиями, переустановками, конфликтами.

2. **Масштабирование упрощается** — Нужно обработать в 2 раза больше запросов? Запустите 2 контейнера вместо одного. Kubernetes (будущее) управляет контейнерами автоматически.

3. **Отсутствие проблем "чистоты"** — Контейнер — изолированный мир; приложение не влияет на другие приложения на сервере. Удалили контейнер? Ничего лишнего на сервере не осталось.

4. **CI/CD и автоматизация** — GitHub Actions может автоматически собрать образ, протестировать, запустить на сервере. Один Dockerfile = весь процесс описан.

5. **Security (безопасность)** — Контейнер не имеет доступ к системе "хоста". Даже если приложение взломали, остальная система в безопасности.

### Как работает Dockerfile: пошаговый разбор

```
# Базовый образ: ОС + Node.js 18
FROM node:18-alpine

# Устанавливаем рабочую директорию внутри контейнера
WORKDIR /app

# Копируем package.json и package-lock.json
COPY package*.json ./

# Устанавливаем зависимости (npm install)
RUN npm install

# Копируем весь код приложения
COPY . .

# Собираем приложение (например, Next.js build)
RUN npm run build

# Открываем порт 3000
EXPOSE 3000

# Команда запуска
CMD ["npm", "start"]
```

**Что происходит:**
1. Берём готовый образ Node.js 18 (слой 1).
2. Устанавливаем рабочую папку (слой 2).
3. Копируем зависимости (слой 3).
4. Устанавливаем их (слой 4).
5. Копируем код (слой 5).
6. Собираем проект (слой 6).
7. Открываем порт (слой 7).
8. Запускаем приложение (слой 8).

Каждый слой кэшируется. Если вы поменяли только код (слой 5), Docker не переустанавливает зависимости (слой 2-4) — экономит время.

### Типы образов для вайбкодинга

**1. Development образ (для разработки)**

```
FROM node:18
WORKDIR /app
COPY package*.json ./
RUN npm install
# Копируем ВСЕ файлы, включая то, что в .gitignore (dev-зависимости, конфиги)
COPY . .
# Запускаем dev-сервер с hot-reload
CMD ["npm", "run", "dev"]
```

Особенность: включает ESLint, тесты, девелопер-инструменты. Образ "толстый", но удобен для разработки.

**2. Production образ (для сервера)**

```
# Stage 1: Сборка
FROM node:18 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
RUN npm run build

# Stage 2: Runtime (только необходимое)
FROM node:18-alpine
WORKDIR /app
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/package.json ./
EXPOSE 3000
CMD ["node", "dist/index.js"]
```

Особенность: Multi-stage build. Сначала собираем всё (с dev-зависимостями), потом копируем только готовый код. Образ "худой": 100-500MB вместо 1-2GB.

### Промпты для AI при Докеризации

**Промпт 1: Базовая Докеризация**

"Моё приложение: Next.js + Node.js бэкенд + PostgreSQL. Создай Dockerfile для production с multi-stage build, .dockerignore для оптимизации размера, и docker-compose.yml для разработки с hot-reload, чтобы я мог разрабатывать локально как обычно."

**Промпт 2: Оптимизация размера образа**

"Мой Docker образ весит 1.5GB, это медленно грузится на сервер. Как оптимизировать? Используй alpine, сжимай слои, удаляй кэши. Дай Dockerfile с лучшими практиками."

**Промпт 3: Для приложения с базой данных**

"Создай docker-compose с моим Next.js приложением, PostgreSQL базой, Redis кэшем. Используй environment variables, volumes для персистентности базы, и сделай так, чтобы приложение автоматически переподключалось при перезапуске БД."

### Общие ошибки при Докеризации

**Ошибка 1: Копировать .env прямо в Dockerfile**

❌ Неправильно:
```
COPY .env .
ENV DATABASE_PASSWORD=secret123
```
Секреты будут видны в образе!

✅ Правильно:
```
# Dockerfile НЕ содержит секреты
# Передаём через docker-compose или переменные окружения
```

**Ошибка 2: Не использовать .dockerignore**

❌ Неправильно:
```
COPY . .  # Копирует всё: .git, node_modules, тесты (4GB)
```

✅ Правильно:
```
# .dockerignore
.git
node_modules
.next
dist
.env
tests
.dockerignore
```

**Ошибка 3: Запускать приложение от root**

❌ Неправильно:
```
USER root
CMD ["npm", "start"]
```

✅ Правильно:
```
RUN useradd -m -u 1000 appuser
USER appuser
CMD ["npm", "start"]
```

**Ошибка 4: Не использовать volumes в docker-compose для разработки**

❌ Неправильно:
```
services:
  web:
    build: .
    # Нет volumes — при изменении кода нужно пересобирать образ
```

✅ Правильно:
```
services:
  web:
    build: .
    volumes:
      - .:/app  # Весь код синхронизируется в реальном времени
      - /app/node_modules  # node_modules не перезаписываются
```

Общие рекомендации — просим AI упаковать всё наше приложение в докер и настроить полную настройку докера чтобы при переносе на сервер легко развернуть наше приложение через докер.

## 3. Типы хостингов и серверов

### Shared Hosting (Общий хостинг)

**Что это:** Ваш сайт живёт на сервере вместе с сотнями других сайтов.
**Плюсы:** Дёшево (2-10$/месяц), не нужно ничего настраивать
**Минусы:** Медленно, ограничения, нет контроля, не подходит для Docker
**Когда выбирать:** Только для простых лендингов на WordPress

### VPS (Virtual Private Server)

**Что это:** Виртуальная машина — часть большого сервера выделена только вам.
**Плюсы:** Полный контроль, root-доступ, можно устанавливать Docker и что угодно, идеально для вайбкодинга
**Минусы:** Нужно настраивать самому (но AI поможет)
**Цена:** 5-20$/месяц для стартовых проектов
**Когда выбирать:** Для большинства веб-приложений — это лучший выбор для новичков

### Cloud Platforms (PaaS)

**Примеры:** Vercel, Netlify, Railway, Render
**Что это:** Платформы, где вы просто загружаете код или коннектите GitHub — они сами всё разворачивают в Docker контейнерах.
**Плюсы:** Максимально просто, автоматический деплой из GitHub, бесплатные тарифы, не нужно думать об инфраструктуре
**Минусы:** Дороже при росте, меньше контроля
**Когда выбирать:** Если хотите максимальную простоту, не хотите возиться с сервером, или начинаете с очень скромного бюджета

### Dedicated Server (Выделенный сервер)

**Что это:** Целый физический сервер только для вас.
**Когда выбирать:** Только для очень крупных проектов (100$/месяц+)
**Вывод для новичков:** Не нужен на старте

## 5. Операционные системы для серверов

### Ubuntu Server (20.04, 22.04, 24.04)

**Рекомендация: Абсолютно лучший выбор для новичков**

Почему:
- Самая популярная — больше всего инструкций, туториалов, помощи
- Стабильная и простая
- Отлично работает с Docker (Docker создан для Linux)
- AI-ассистенты лучше всего знают команды для Ubuntu
- LTS версии поддерживаются 5 лет (22.04 LTS до 2027 года)

**Версия для выбора:**
- 22.04 LTS ← Выбирайте эту (поддержка до 2027)
- 24.04 LTS (новая, поддержка до 2029, но меньше примеров в интернете)

**Промпт для AI:**
"Помоги настроить Ubuntu 22.04 LTS VPS для деплоя Docker приложения: установка Docker, docker-compose, nginx, firewall, создание пользователя. Дай пошаговые команды."

### Debian

**Что это:** "Родитель" Ubuntu, более консервативная система, очень стабильна.
**Для новичков:** Не рекомендуется — команды немного отличаются от Ubuntu, меньше туториалов

### CentOS / AlmaLinux / Rocky Linux

**Что это:** Альтернативная линейка серверных Linux, популярны в больших компаниях.
**Для новичков: Избегайте** — другие команды (yum вместо apt), меньше материалов на русском, более сложная для начинающих

### Windows Server

**Для новичков: Не рассматривайте вообще**
- Дорого ($100+/месяц)
- Docker на Windows сложнее, работает медленнее
- Меньше поддержки Docker
- Не стандарт для веб-разработки

**Финальный вывод по ОС:**
Используйте Ubuntu 22.04 LTS — это фактический стандарт индустрии для веб-приложений и Docker.

## 6. Ресурсы сервера: как выбрать железо

### CPU (Процессор)

**Что это:** Количество ядер процессора. Каждое ядро может обрабатывать 1 задачу одновременно.

**Рекомендации:**
- 1 CPU (1 ядро): Совсем слабый, для тестирования только
- 2 CPU (2 ядра): Хороший выбор для стартапа с 100-1000 пользователей
- 4 CPU (4 ядра): Для приложения с 1000-10000 пользователей, или если много обработки данных
- 8+ CPU: Для крупных проектов

Правило: 2 CPU на старте, потом мониторьте нагрузку (htop) и увеличивайте при нужде.

### RAM (Оперативная память)

**Критически важно!** Это главное узкое место на дешёвых хостингах.

**Что нужно:**
- 1GB: Только для очень лёгких приложений (статические сайты, малый Nuxt)
- 2GB: Node.js + Frontend (хороший выбор для стартапа)
- 4GB: Node.js + PostgreSQL + Redis (рекомендуется для реального приложения)
- 8GB: Для тяжёлых проектов, ML, большие базы данных
- 16GB+: Крупные приложения, большой кэш

**Почему важно:** Если RAM кончается, операционная система начинает использовать диск (SWAP) — это в 1000 раз медленнее. Приложение ползет, или падает с ошибкой "Out of Memory".

**Мониторинг в реальном времени:**
```
# На сервере
free -h  # Показать свободную память
top  # Живой мониторинг использования
```

### Disk (Диск)

**Типы:**
- HDD (жёсткий диск): Медленный, архаичный — избегайте
- SSD (solid state drive): Быстрый — всегда выбирайте это
- NVMe SSD: Очень быстрый (в 3-5 раз быстрее SSD) — если стоит не намного дороже, берите

**Объём:**
- 20GB: Совсем мало, только для tiny приложений
- 50GB: Хороший стартовый объём для Node.js + PostgreSQL
- 100GB: Если хранить файлы пользователей, логи, бэкапы
- 200GB+: Большие базы данных, много медиа

Правило выбора: 50GB SSD для старта. Потом мониторьте df -h (свободное место), при 80% заполнения расширяйте.

### Bandwidth (Пропускная способность)

**Что это:** Сколько данных может "вытечь" из сервера в интернет за месяц.

**На практике:**
- 1 TB/месяц: Хватит для 100,000 пользователей с просмотром видео
- 2 TB/месяц: Стандартный выбор
- Неограниченный: Обычно означает "справедливое использование" (если не качаете торренты 24/7)

Редко это проблема: Большинство хостингов не ограничивают или ограничивают очень мягко.

### Конфигурация сервера: Рекомендации

**Конфиг 1: Тестирование/Учёба (Самый дешёвый)**
- 1 CPU
- 1-2GB RAM
- 25GB SSD
- Примеры: Selectel 300₽, DigitalOcean $6/месяц
- Годится для: Учёбы, MVP, личного проекта
- Не годится для: Production с реальными пользователями

**Конфиг 2: Стартап/MVP (Рекомендуется для новичка, плюс сможете запустить в Cursor)**
- 2 CPU
- 4GB RAM ← Важно! а лучше 6-8 и с экономить на памяти ее хватит и 20-30 на первых парах.
- 50GB SSD

### Пошаговая инструкция подключения к своему VPS через Cursor, VS Code и др.

У вас есть:
- IP адрес сервера, логин и пароль из письма провайдера для первого входа по SSH.
- Локальная машина с установленным OpenSSH клиентом и VS Code с расширением Remote-SSH (установим в шагах).

bash - это означает что команду выполняем в терминале

**Шаг 1. Сгенерируйте SSH-ключ на локальной машине**

Рекомендуемый тип: ed25519, он современный и компактный, подходит для большинства задач.

Команда macOS/Linux:
```bash
ssh-keygen -t ed25519 -C "ваш-комментарий" -f ~/.ssh/id_ed25519
```

В процессе введите при желании надёжную passphrase для защиты приватного ключа, публичный ключ появится в ~/.ssh/id_ed25519.pub.

На Windows используйте ту же команду в PowerShell или Git Bash, ключи сохранятся в C:\Users\Имя\.ssh\id_ed25519(.pub) по аналогии с Unix.

**Шаг 2. Загрузите публичный ключ на сервер**

Вариант А — автоматом через ssh-copy-id (macOS/Linux или Windows Git Bash):

```bash
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@SERVER_IP
```

Вас попросят пароль от сервера один раз, утилита сама создаст ~/.ssh, добавит ключ в authorized_keys и выставит корректные права.

Вариант B — вручную, если ssh-copy-id недоступен:
a. Скопируйте содержимое вашего публичного ключа: cat ~/.ssh/id_ed25519.pub и сохраните буфер обмена.
b. Зайдите на сервер по паролю: ssh user@SERVER_IP и в сессии создайте каталог и файл ключей с правами.

```bash
mkdir -p ~/.ssh && chmod 700 ~/.ssh
echo "ВСТАВЬТЕ_СЮДА_ОДНОЙ_СТРОКОЙ_СОДЕРЖИМОЕ_.pub" >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys
```

**Шаг 3. Проверьте вход по ключу из терминала**

```bash
ssh -i ~/.ssh/id_ed25519 user@SERVER_IP
```

Если всё ок, будет запрос только passphrase от ключа (если задавали), а не пароль учётной записи сервера.

**Шаг 4. Установите и настройте VS Code Remote-SSH**

Установите расширение Remote-SSH в VS Code через Marketplace, после установки появится пункт "Remote-SSH" и меню подключения в статус-баре.

Откройте палитру команд и выберите "Remote-SSH: Add New SSH Host...", затем вставьте команду подключения с ключом, например: ssh user@SERVER_IP -i ~/.ssh/id_ed25519, и укажите, куда сохранить запись (обычно ~/.ssh/config).

Теперь выберите "Remote-SSH: Connect to Host...", кликните на созданный хост и при первом подключении подтвердите fingerprint и платформу "Linux", дождитесь установки VS Code Server на удалённой машине.

**Шаг 5. Откройте папку проекта и терминал на сервере**

После установления соединения нажмите "Open Folder" и укажите каталог на сервере (например, /srv/app), чтобы работать с проектом удалённо как с локальным.

Встроенный терминал VS Code (Terminal → New Terminal) теперь выполняет команды на сервере от вашего пользователя, что удобно для docker compose, git и сервисных команд.

**Шаг 6. Частые ошибки и быстрые решения**

- Permission denied (publickey): проверьте права на приватный ключ (chmod 600 ~/.ssh/id_ed25519), на ~/.ssh (chmod 700 ~/.ssh) и на файл authorized_keys (chmod 600), затем повторите подключение.
- Подключение идёт "не тем" ключом: при добавлении хоста в VS Code указывайте явный -i ~/.ssh/id_ed25519, либо отредактируйте запись хоста через "Remote-SSH: Open SSH Configuration File..." и добавьте IdentityFile для нужного ключа.
- Windows: убедитесь, что установлен OpenSSH Client, иначе VS Code не сможет вызвать системный ssh, после установки перезапустите VS Code и повторите подключение.

### Шаг 6: Настройка Nginx как reverse proxy

Зачем: Nginx принимает запросы на порты 80 (HTTP) и 443 (HTTPS) и перенаправляет их вашему приложению на порт 3000.

**6.1 Создание конфигурации Nginx**

Попросите AI:
"Создай конфиг Nginx для проксирования на мое Node.js приложение, работающее на localhost:3000, с доменом example.com. Добавь также www версию (www.example.com). Включи поддержку WebSocket (для Socket.io если нужно)."

AI вернёт примерно такое:

```
Файл: /etc/nginx/sites-available/example.com

upstream app {
  server localhost:3000;
}

server {
  listen 80;
  server_name example.com www.example.com;

  location / {
    proxy_pass http://app;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection 'upgrade';
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_cache_bypass $http_upgrade;
  }
}
```

**6.2 На сервере, создайте файл:**

```bash
# Переходим в папку конфигов Nginx
cd /etc/nginx/sites-available
# Создаём конфиг (замените example.com на ваш домен)
sudo nano example.com
```
Вставляете текст от AI (Ctrl+Shift+V).
Выходите: Ctrl+X → Y → Enter

**6.3 Активируете конфиг**

```bash
# Создаёте симлинк в sites-enabled
sudo ln -s /etc/nginx/sites-available/example.com /etc/nginx/sites-enabled/
# Проверяете синтаксис
sudo nginx -t
# Должно быть: "successful"
# Перезагружаете Nginx
sudo systemctl restart nginx
```

**6.4 Проверяете работу**

```bash
# Смотрите, что Nginx запущен
sudo systemctl status nginx
# Проверяете, что порт 80 слушается
sudo ss -tlnp | grep nginx
```

### Шаг 8: Настройка SSL/HTTPS (очень важно!)

**Что такое SSL/HTTPS в простых словах:**

HTTP = "Почта без конверта" — письмо открытое, все видят содержимое. Данные передаются простым текстом. Хакер может перехватить пароль, данные карты. Браузер показывает: "⚠ Не защищено"

HTTPS = "Почта в запечатанном конверте" — только адресат может прочитать. Данные шифруются перед отправкой. Хакер видит только зашифрованный текст (бессмыслица). Браузер показывает: "🔒 Защищено"

SSL/TLS сертификат = "Печать на конверте" — доказывает что это именно ваш сайт. Выдаётся удостоверяющим центром (Certificate Authority). Проверяет, что домен example.com действительно ваш. Браузер проверяет сертификат и показывает 🔒

**Зачем нужно HTTPS:**
- ✅ Безопасность: пароли пользователей не видны хакерам
- ✅ SEO: Google ранкирует HTTPS сайты выше
- ✅ Доверие: пользователи видят 🔒 и доверяют
- ✅ Требование: PayPal, Google, мобильные приложения требуют HTTPS
- ✅ Закон: GDPR требует HTTPS для обработки данных

Let's Encrypt = Бесплатный сертификат, автоматически обновляется каждые 90 дней

**8.1 Установка Certbot (инструмент для Let's Encrypt)**


```bash

```

Что происходит:
- Certbot проверяет, что домен указывает на ваш сервер
- Let's Encrypt выдаёт сертификат
- Certbot автоматически обновляет конфиг Nginx для HTTPS
- Certbot настраивает автоматическое обновление сертификата

Вам нужно ответить на вопросы:
```

Do you agree to the ACME server terms of service? (Y/n) y
```

### Docker: Лучшие практики и оптимизация

**Multi-stage build: Уменьшаем размер образа**

```dockerfile
# Stage 1: Builder
FROM node:18 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production && npm cache clean --force
COPY . .
RUN npm run build

# Stage 2: Runtime (только то что нужно для запуска)
FROM node:18-alpine  # alpine в 5 раз меньше!
WORKDIR /app
# Копируем только готовый код из builder
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/package.json ./
EXPOSE 3000
CMD ["node", "dist/index.js"]
```

Размер:
- Без multi-stage: 1.5GB
- С multi-stage: 200-500MB
- Загружается в 3 раза быстрее!

**Кэширование слоёв: Ускоряем сборку**

❌ Медленно: при любом изменении переустанавливает всё
```dockerfile
FROM node:18
WORKDIR /app
COPY . .  # Копируем всё
RUN npm install  # Если код изменился, переустанавливает
```

✅ Быстро: переустанавливает только если package.json изменился
```dockerfile
FROM node:18
WORKDIR /app
COPY package*.json ./  # Копируем зависимости ОТДЕЛЬНО
RUN npm install  # Кэшируется на этом шаге
COPY . .  # Копируем код ПОСЛЕ install
RUN npm run build
```

На сборку:
- При изменении кода: 5 секунд
- При изменении зависимостей: 60 секунд

**Health checks: Проверяем живо ли приложение**

```yaml
# docker-compose.yml
services:
  web:
    build: .
    ports:
      - "3000:3000"
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s
```

Docker будет проверять каждые 30 секунд, живо ли приложение.
