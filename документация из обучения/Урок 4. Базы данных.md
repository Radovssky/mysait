**Страница 2**

Основные Понятия: Что Такое Данные?

Данные — это информация, которую хранит ваше приложение. Но не все данные организованы одинаково. Понимание различных типов структур данных — это первый шаг к проектированию хорошей базы данных.

Структурированные Данные

Структурированные данные организованы по чётким схемам — как таблица в Excel. Каждый элемент знает, где он находится и какого он типа. Примеры: пользователи в системе, заказы товаров, информация о продуктах.

Плюсы:
Легко искать и фильтровать данные
Безопасность и контроль доступа
Быстрый анализ информации
Гарантия консистентности

Минусы:
Нужно определить схему заранее
Сложно менять структуру потом
Может быть излишним для простых данных
Требует планирования

Неструктурированные и Полуструктурированные Данные

Неструктурированные данные не имеют фиксированной схемы — текстовые комментарии, фотографии, видео. Полуструктурированные данные (как JSON) имеют некоторую структуру, но она гибкая: поля могут отличаться между записями. Они идеальны для быстрого прототипирования и приложений, где структура часто меняется.

**Страница 3**

Типы Данных: Строительные Блоки Информации

В любой базе данных данные организуются по определённым типам. Понимание этих типов критически важно для правильного проектирования схемы и эффективного хранения информации. Каждый тип данных предназначен для определённой цели и имеет свои особенности.

Числа (Numbers)
Используются для количественных данных, расчетов и уникальных идентификаторов. Могут быть целыми (integer) или с плавающей точкой (float/decimal).
123
3.14159
-42

Строки (Strings)
Представляют текстовую информацию, такую как имена, описания, адреса или комментарии. Могут иметь переменную длину.
"Hello, World!"
"email@example.com"
"Product Description"

Булевы (Booleans)
Логические значения, представляющие истину (true) или ложь (false). Часто используются для флагов, статусов или условий.
true
false

Даты и Время (Dates & Times)
Используются для записи моментов времени, событий, сроков действия или временных меток создания/изменения записей.
2023-10-27 (только дата)
14:30:00 (только время)
2023-10-27T14:30:00Z (дата и время с часовым поясом)

Массивы (Arrays)
Упорядоченные коллекции элементов. Могут хранить несколько значений одного или разных типов, например, список тегов или ID пользователей.
[1, 2, 3, 4, 5]
["apple", "banana", "cherry"]

Объекты (Objects/JSON)
Структурированные наборы данных, состоящие из пар "ключ-значение". Позволяют хранить сложные иерархические структуры, как, например, профиль пользователя с множеством полей.
{
  "name": "Иван",
  "age": 30,
  "isActive": true
}

Перечисления (Enums)
Специальный тип данных, который ограничивает значение поля предопределенным набором возможных вариантов. Это помогает обеспечить целостность данных и избежать опечаток.
// Допустимые значения для поля "status":
// "pending", "approved", "rejected"
// Пример использования:
status: "approved"

**Страница 4**

Где и Как Хранятся Данные на Фронтенде

Перед тем как отправлять данные на сервер, приложение может хранить информацию локально в браузере пользователя. Понимание различных способов хранения критически важно для создания быстрых и отзывчивых приложений.

1. State (Состояние в памяти)
Данные в оперативной памяти браузера. Теряются при перезагрузке. Идеально для текущего состояния UI, значений в инпутах, открытых модальных окон.
Когда используется:
Текущее состояние UI (открыт ли модальное окно?)
Временные данные (значение в инпуте)
Данные, которые нужны только сейчас
Плюсы:
Очень быстро (в памяти)
Простой синтаксис
Идеально для UI состояния
Минусы:
Теряется при перезагрузке страницы
Видно в браузере (безопасность!)
Ограничено памятью браузера

2. LocalStorage
Встроенное хранилище браузера (~5-10 МБ). Сохраняется между сеансами. Идеально для настроек пользователя, кеша, черновиков. Синхронный доступ.
Когда используется:
Сохранённые настройки пользователя (тема, язык)
Кеш данных (чтобы не запрашивать сервер каждый раз)
Черновики (пользователь заполнил форму, потом закрыл, потом вернулся)
Токены авторизации (в некоторых случаях)
Плюсы:
Сохраняется между сеансами
До ~5-10 МБ на браузер
Синхронный доступ
Простой API
Минусы:
Не безопасен (видно в DevTools)
Медленнее чем State
Не работает с большими данными
Синхронный (может заблокировать UI)

3. SessionStorage
Похож на LocalStorage, но удаляется при закрытии браузера. Используется для временных данных текущего сеанса. Синхронный доступ.
Когда используется:
Временные данные в текущем сеансе
Данные, которые не должны переживать закрытие браузера

4. IndexedDB
Мощная клиентская база данных (~50+ ГБ). Идеально для кеша больших объёмов данных, offline-приложений, PWA. Асинхронный доступ, поддерживает индексы.
Когда используется:
Кеш больших количеств данных (например, список всех товаров)
Offline-first приложения (работают без интернета)
PWA (Progressive Web Apps)
Хранение медиа данных (картинки, видео)
Плюсы:
До 50+ GB на браузер (!)
Асинхронный (не блокирует UI)
Индексы для быстрого поиска
Транзакции (гарантирует целостность)
Минусы:
Сложнее API (нужны Promises или async/await)
Медленнее, чем State
Может быть очищено браузером при нехватке места
Требует setup для работы с индексами

5. Cookies
Маленькие куски данных (~4 КБ), отправляются на сервер. Используются для авторизации, сессий, аналитики. Обязательно используйте httpOnly для защиты токенов!
Типы cookies:
Session Cookies — удаляются при закрытии браузера
Persistent Cookies — остаются до истечения срока
Secure Cookies — отправляются только по HTTPS
HttpOnly Cookies — не доступны JavaScript (безопаснее для токенов!)
SameSite Cookies — защита от CSRF атак

Важно: Никогда не сохраняйте пароли или конфиденциальные токены в LocalStorage! Используйте httpOnly cookies для максимальной безопасности.

**Страница 5**

Типы Баз Данных: Реляционные (SQL)

Реляционные базы данных хранят информацию в таблицах, похожих на электронные таблицы. Это наиболее надёжный и проверенный способ хранения структурированных данных, используемый в банках, социальных сетях и e-commerce.

PostgreSQL: Лучший Выбор для Большинства Проектов

PostgreSQL — это мощная, надёжная, open-source база данных, используемая Spotify, Instagram и Netflix. Она предлагает надёжность (ACID транзакции), мощный SQL язык и даже встроенную поддержку JSON.

Надёжность — ACID транзакции гарантируют целостность данных даже при сбоях
Производительность — Быстрые запросы с индексами и оптимизированным движком
Масштабируемость — Справляется с миллионами строк и сложными запросами
Безопасность — Встроенная система прав доступа и шифрование

MySQL и MariaDB

MySQL проще для начинающих, но менее мощная. MariaDB — это улучшенный fork MySQL с лучшей функциональностью. Выбирайте PostgreSQL для серьёзных проектов.

Ключевые Концепции SQL

Primary Key (первичный ключ) — уникальный идентификатор каждой строки. Foreign Key (внешний ключ) — ссылка на Primary Key другой таблицы, гарантирует целостность связей. Constraints (ограничения) — правила для данных (UNIQUE, NOT NULL, CHECK). Индексы — ускорят поиск данных (как оглавление в книге).

Пример psql запросов:
CREATE TABLE users (id SERIAL PRIMARY KEY, email VARCHAR(100) UNIQUE NOT NULL, name VARCHAR(100));
INSERT INTO users (email, name) VALUES ('ivan@example.com', 'Ivan');
SELECT * FROM users;
UPDATE users SET name = 'Ivanov' WHERE id = 1;

Сетевые vs Встраиваемые Базы Данных

Сетевые БД:
Работают как отдельный процесс/сервис
Нужно устанавливать и запускать на ПК/сервере
Подключение через сеть (даже если localhost)
Могут обслуживать множество клиентов одновременно
Требуют настройки портов, пользователей, прав доступа
Примеры: PostgreSQL (порт 5432), MySQL (порт 3306), MongoDB (порт 27017)

Встраиваемые БД:
Встраиваются прямо в приложение
Не требуют отдельной установки сервера
Работают как файл на диске
Один процесс = одна БД
Идеально для мобильных приложений, десктопных программ, прототипов
Примеры: SQLite (файл .db), LevelDB

**Страница 6**

Типы Баз Данных: NoSQL и Специализированные

MongoDB: Гибкость и Простота

MongoDB хранит данные как JSON документы. Нет фиксированной схемы — структура может меняться от документа к документу. Идеально для быстрого прототипирования, контент-систем, социальных сетей где гибкость критична.

Преимущества:
Очень гибкая структура
JSON-подобный формат (естественен для JS)
Быстро разрабатывать
Горизонтальное масштабирование
Встроенные массивы и объекты

Недостатки:
Требует больше памяти и диска
Сложнее с связями данных
Нужны проверки типов при работе
Меньше гарантий консистентности
Медленнее сложные запросы

Firebase/Firestore: Облачная БД в Реальном Времени

Firestore от Google — облачная БД с автоматическим масштабированием и real-time обновлениями. Идеально для мобильных приложений, чатов, совместного редактирования. Не нужно управлять сервером, но платите за операции.

Redis: Молниеносное Key-Value хранилище

Redis хранит данные в памяти сервера. Невероятно быстро! Используется для кеширования (разгрузка основной БД), сессий, счётчиков лайков, очередей задач, рейт-лимитинга. В каждом production приложении есть Redis!

Кеширование — Самое быстрое хранилище результатов запросов
Сессии — Хранение данных текущего пользователя
Счётчики — Лайки, просмотры, переходы по ссылкам
Очереди — Background задачи и асинхронная обработка

Elasticsearch / Meilisearch: Поиск по Тексту

Специализированные БД для быстрого полнотекстового поиска. Elasticsearch более мощная, но сложная. Meilisearch — простая и красивая альтернатива. Используются в Amazon, Google, везде где нужен поиск.

**Страница 7**

Проектирование Схемы Базы Данных

Нормализация: Избегайте Дублирования

Нормализация — это процесс организации данных чтобы избежать дублирования и противоречивости. Плохо спроектированная схема приводит к проблемам: если изменить данные в одном месте, нужно менять везде.

1. Денормализованная схема — Все в одной таблице, много дублирования данных, сложно обновлять
2. Нормализованная схема — Данные разбиты по отдельным таблицам, каждый факт хранится один раз

Primary Key и Foreign Key

Primary Key — уникальный идентификатор каждой строки (обычно просто ID). Foreign Key — ссылка на Primary Key другой таблицы. Вместе они создают связи между таблицами и гарантируют целостность данных.

Типичные Связи между Таблицами

One-to-Many (1:M)
Один пользователь → много заказов. Самая частая связь. Решается через Foreign Key.

Many-to-Many (M:M)
Один товар → много заказов, один заказ → много товаров. Решается через промежуточную таблицу.

One-to-One (1:1)
Один пользователь → одна аватарка. Редкая связь. Решается через UNIQUE Foreign Key.

Индексы: Ускорение Поиска

Индекс — это ускоренный способ найти данные (как оглавление в книге). Создавайте индексы на полях, которые часто используются в WHERE, JOIN и ORDER BY. Но не переусложняйте: индексы занимают место и замедляют INSERT, UPDATE, DELETE.

**Страница 8**

Масштабирование и Оптимизация БД

Проблема N+1 Query: Кошмар Разработчика

N+1 Query Problem — это когда вы делаете 1 основной запрос и потом N дополнительных запросов в цикле. Результат: 1001 запрос вместо 1. Это убивает производительность приложения!

❌ Плохо: N+1 Queries
Запрос: все пользователи
Потом для каждого пользователя: запрос их заказов
1000 пользователей = 1001 запрос!

✅ Хорошо: JOIN
Один запрос со JOIN объединяет таблицы
Получаем пользователей с их заказами
Ровно 1 запрос вне зависимости от количества

Sharding: Разделение Данных

Sharding — это разделение данных между несколькими серверами БД. Нужно когда данных очень много (100+ ГБ, миллионы пользователей). Пользователи с ID 1-1000000 на сервере 1, 1000001-2000000 на сервере 2, и т.д. Очень сложная архитектура, но позволяет масштабироваться бесконечно.

Репликация: Отказоустойчивость и Производительность

Репликация — копирование данных на несколько серверов. Master сервер получает все записи. Replicas копируют данные и могут использоваться для чтения. Если Master падает, один из Replicas становится Master. Используется везде на production.

Master — Основной сервер, записи идут сюда
Replica 1 — Копия для чтения, на случай отказа
Replica 2 — Ещё одна копия для надёжности

**Страница 9**

Шифрование и Безопасность БД

Три Уровня Защиты

Шифрование в Пути (In Transit)
Все данные между клиентом и сервером проходят через HTTPS. Даже если кто-то перехватит трафик, не сможет прочитать из-за шифрования TLS/SSL.

Шифрование в Покое (At Rest)
Данные на диске сервера зашифрованы. Если кто-то украдёт физический сервер, данные будут защищены. Используется шифрование на уровне ОС (LUKS, BitLocker).

Шифрование Чувствительных Полей
Особо конфиденциальные данные (номера карт, SSN, паспорта) шифруются индивидуально даже внутри БД. Используйте AES-256 с защитой ключей.

Хеширование Паролей: Никогда не Сохраняйте в Открытом Виде

Критически важно: Пароли ДОЛЖНЫ быть захеширована. Используйте bcrypt — она медленная (защита от brute-force), адаптивная (становится медленнее каждый год) и имеет встроенную соль.

Как Это Работает:
1. Пользователь вводит пароль "secret123"
2. Вычисляем bcrypt хеш (невозможно вернуть пароль!)
3. Сохраняем только хеш в БД
4. При логине сравниваем новый хеш со сохранённым
5. Если совпадают → пароль правильный

Результат: Даже если хакер украдёт БД, пароли остаются защищены. Невозможно узнать пароль из хеша.

Защита Токенов и API Ключей

НИКОГДА не сохраняйте токены в LocalStorage — это видно в DevTools! Используйте httpOnly cookies — они недоступны JavaScript и защищены от XSS атак. Для особо важных токенов используйте SameSite attribute для защиты от CSRF.

Безопасность в VibeCooding: Соблюдайте все три уровня защиты. Шифрование в пути обязательно (HTTPS везде!). Хешируйте пароли bcrypt. Для чувствительных данных используйте дополнительное шифрование. Проверяйте права доступа на всех уровнях.

Типичная Stack Приложения

Frontend Layer — State, LocalStorage, IndexedDB для локального хранилища в браузере

**Страница 11**

Дополнительные Инструменты и Хранение Медиа

ORM: Object-Relational Mapping (Объектно-реляционное отображение)

ORM — это прослойка между вашим кодом (объектами) и базой данных (таблицами). Вместо написания SQL-запросов, вы работаете с классами и объектами, а ORM преобразует эти операции в SQL. Это упрощает разработку, делает код более чистым и переносимым.

Абстракция — Работайте с объектами вашего языка программирования, а не с SQL-таблицами.
Продуктивность — Сокращает объем boilerplate кода и ускоряет разработку приложений.
Типовая Безопасность — Многие ORM обеспечивают проверку типов данных на уровне кода, уменьшая ошибки.

Миграции Базы Данных и Версионирование

Миграции — это контролируемые изменения в схеме вашей базы данных. Они позволяют отслеживать, применять и откатывать изменения схемы БД (добавление таблиц, столбцов, изменение типов данных) как часть вашего кода.

Отслеживание Изменений — Сохраняется полная история всех изменений в схеме БД.
Совместная Работа — Облегчает работу команды над одной и той же базой данных.
Откат — Возможность легко вернуться к предыдущему состоянию схемы в случае проблем.

Хранение Медиа (Изображения, Видео, Аудио)

Изображения, видео и аудио файлы обычно не хранятся напрямую в реляционных базах данных из-за их размера, что может замедлить работу БД и увеличить её стоимость. Вместо этого в БД хранится ссылка (URL или путь) на файл, а сами файлы размещаются во внешнем хранилище.

Для хранения медиа используются специализированные сервисы объектного хранения, такие как Amazon S3, Google Cloud Storage или MinIO. Эти сервисы оптимизированы для эффективного хранения и быстрой отдачи больших объемов файлов. В базе данных достаточно хранить лишь путь или уникальный идентификатор файла, чтобы иметь к нему доступ.

Backend возвращает фронтенду подтверждение (обычно JSON с новыми данными). Фронтенд обновляет UI.

Ключевые Навыки Вайбкодера

Проектирование Схемы — Уметь спроектировать структуру БД так чтобы не было дублирования и запросы были быстрыми.
SQL / Query Язык — Писать эффективные запросы(промты), JOIN'ы, использовать индексы, избегать N+1 problem.
Безопасность — Защита от SQL injection, хеширование паролей, шифрование данных, правильное управление правами доступа.
Оптимизация — Кеширование, индексы, выбор правильной БД для задачи, мониторинг производительности.

Масштабирование — Когда приложение растёт: sharding, репликация, выбор между SQL и NoSQL.

Частые Ошибки и Как их Избежать

❌ Сохранение Паролей в Открытом Виде
Результат: любой кто получит доступ к БД, получит все пароли. Решение: Хешируйте с bcrypt!

❌ N+1 Queries в Цикле
Результат: приложение тормозит. Решение: Используйте JOIN, batch queries или eager loading.

❌ Отсутствие Индексов
Результат: запросы медленнеют при большом количестве данных. Решение: Создавайте индексы на часто используемые поля.

❌ Хранение Больших Данных в LocalStorage
Результат: браузер тормозит, может заблокироваться. Решение: Используйте IndexedDB для больших объёмов.

❌ Отсутствие Кеширования
Результат: каждый запрос идёт в БД (медленно). Решение: Используйте Redis для часто запрашиваемых данных.

❌ Передача Пароля в URL или GET параметрах
Результат: пароль видно в истории браузера и логах. Решение: Используйте POST, передавайте через тело запроса.

Пример установки базы данных на ваш пк через Docker

Что такое Docker и зачем он тебе — это инструмент, который позволяет запускать приложения в изолированных контейнерах. Контейнер — это как мини-компьютер внутри твоего компьютера, где можно установить всё, что нужно: базу данных, сервер, зависимости.

Зачем использовать Docker:
Удобно разворачивать и удалять окружение
Нет конфликтов между версиями библиотек
Можно легко делиться конфигурацией с другими

Что такое WSL и зачем он нужен — WSL (Windows Subsystem for Linux) — это способ запускать Linux прямо внутри Windows. Docker на Windows работает через WSL, потому что контейнеры используют Linux.

WSL нужен для:
Запуска Docker Engine
Использования Linux-команд и утилит
Упрощения работы с контейнерами

Как всё установить и настроить (для Windows) (для другой операционной системы настройка проще, для получения информации попросить ИИ объяснить как пошагово добавить docker в вашу операционную систему какие настройки нужно выполнить и как подключить ide для работы с докером)

1. Установи WSL. Открой PowerShell от имени администратора. Выполни команду: wsl --install. Перезагрузи компьютер. По умолчанию установится Ubuntu.

2. Установи Docker Desktop. Скачай с официального сайта Docker. Установи и при установке включи интеграцию с WSL. После установки убедись, что Docker работает (иконка в трее).

3. Настрой Docker. Открой Docker Desktop → Settings → Resources → WSL Integration. Включи интеграцию с твоим дистрибутивом (например, Ubuntu).

Подключение IDE (например, VS Code)

Вариант 1: VS Code + Remote Containers. Установи VS Code. Установи расширение Remote - Containers. Открой папку проекта → нажми Ctrl+Shift+P → выбери Remote-Containers: Open Folder in Container. VS Code подключится к контейнеру, и ты сможешь работать как будто это обычная среда.

Вариант 2: Подключение к базе из IDE. Запусти контейнер с базой данных (например, PostgreSQL). В IDE (например, DataGrip, DBeaver, VS Code с SQLTools) укажи:
Хост: localhost
Порт: 5432 (или другой, если менял)
Логин/пароль: как указано в docker-compose.yml

Что дальше? Ты можешь:
Добавить pgAdmin для визуального управления
Использовать другие базы: MySQL, MongoDB, Redis
Создать Dockerfile для своего приложения
